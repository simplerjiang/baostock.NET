# Baostock.NET v1.2.0 设计与实施计划

> 文档状态：DRAFT  
> 版本：v1.2.0  
> 适用 SDK：Baostock.NET（.NET 8 / C# 12）

---

## 1. 版本目标

v1.2.0 是 Baostock.NET 自项目启动以来的**首次重大功能扩展**，目标是把当前「baostock 单源 TCP 客户端」升级为「**A 股综合数据 SDK**」，同时保持对老用户的克制升级路径（仅一次代码格式破坏，其它 API 形态保留）。

具体目标：

1. **保留**现有 baostock TCP 协议 27 个 API 的全部能力（登录、K 线、复权因子、行业、成分股、财务、宏观等），不丢失任何一个入口。
2. **扩展**为多源 SDK，新增覆盖：
   - 实时行情（腾讯 / 新浪 / 东方财富）
   - 历史 K 线（东方财富 push2his、可对照腾讯）
   - 财务报表三表（东方财富 / 新浪）
   - 券商研报、十大股东、龙虎榜、宏观（GDP/CPI/PMI 等）
   - 巨潮资讯网公告 PDF（Stream + 落盘）
3. 引入「**Hedged Requests + 健康感知**」的多源**无感切换**架构：用户调用一次 API，SDK 内部对一组源做对冲请求，任意源先成功即返回；失败源自动进入冷却。
4. 全部对外接口统一返回模型中带 `string DataSource { get; }`，保留可观测性。
5. 完成 README、CHANGELOG、UserAgentTest、单元测试样例的同步更新，并以 `1.2.0` 发版到 NuGet。

非目标（v1.2.0 不做）：

- 不做行情 WebSocket 长连。
- 不做行情写入数据库 / 持久化。
- 不引入 DI / Hosting 集成（保持纯 SDK）。
- 不做指标计算（MACD / KDJ 等）。

## 2. 破坏性变更（BREAKING）

> ⚠️ v1.2.0 **不是** 1.1.x 的纯增量发布，存在**一次性**破坏性变更，必须在 CHANGELOG 顶部以 `⚠️ BREAKING` 显著标注。

### 2.1 对外股票代码格式统一

- **新格式（东方财富风格）**：`SH600000`、`SZ000001`、`BJ430047`
  - 大写交易所前缀（`SH` / `SZ` / `BJ`）
  - 紧跟 6 位数字代码
  - **无点号**、**无空格**
- 该格式同时作为：
  - 全部新增 API（实时行情、东财 K 线、财务、研报、股东、龙虎榜、PDF 等）的入参格式
  - **现有 27 个 baostock TCP API 的入参格式**
- SDK 内部由 `CodeFormatter` 在调用 baostock TCP 协议前翻译为 baostock 原生格式 `sh.600000` / `sz.000001` / `bj.430047`，对用户透明。

### 2.2 影响范围

- 旧用户原本传入 `sh.600000` 形式的代码 **将不再被接受**（会抛 `ArgumentException` 提示使用东财风格）。
- 文档全量更新：
  - `README.md` 所有示例
  - `README.UserAgentTest.md` 所有示例
  - `docs/getting-started.md`、`docs/PROJECT-PLAN.md` 中涉及 code 的片段
  - 所有现有单元测试 / 集成测试样例代码
- `CHANGELOG.md` v1.2.0 段落必须**顶部第一行**写 `⚠️ BREAKING：股票代码格式统一为东方财富风格（SH600000 / SZ000001 / BJ430047）`。

### 2.3 兼容策略

- v1.2.0 不提供「双格式兼容开关」。
- `CodeFormatter` 对明显的旧格式（含点号）抛出**带迁移建议**的异常：

  ```text
  ArgumentException: 股票代码 "sh.600000" 使用了旧格式（baostock 原生）。
  自 v1.2.0 起请改用东方财富风格 "SH600000"。SDK 内部会自动翻译。
  ```

### 3.2 核心抽象

```csharp
namespace Baostock.NET.Http;

/// <summary>
/// 单值数据源契约。每个 (数据类型, 上游) 组合实现一个。
/// </summary>
public interface IDataSource<TReq, TRes>
{
    /// <summary>
    /// 源名称，进入 TRes.DataSource，便于排障。
    /// 例：tencent, sina, eastmoney, cninfo。
    /// </summary>
    string Name { get; }

    /// <summary>
    /// 源链优先级，数值越小越优先（P0 最高）。
    /// </summary>
    int Priority { get; }

    /// <summary>
    /// 拉取并解析。失败抛异常，由 Runner 决定走 hedge / 失败重试 / 标记冷却。
    /// </summary>
    Task<TRes> FetchAsync(TReq request, CancellationToken cancellationToken);
}

/// <summary>
/// 流式数据源契约。用于 PDF / 大文件等不应一次性载入内存的场景。
/// </summary>
public interface IStreamDataSource<TReq, TRes>
{
    string Name { get; }
    int Priority { get; }

    /// <summary>
    /// 返回的 TRes 内部持有 Stream，调用方负责 Dispose。
    /// </summary>
    Task<TRes> FetchAsync(TReq request, CancellationToken cancellationToken);
}
```

### 3.3 HedgedRequestRunner 算法

输入：一组 `IDataSource<TReq,TRes>`、一个 `TReq`、`hedgeInterval`、`SourceHealthRegistry`、`CancellationToken`。

执行步骤：

1. 从源列表挑出**当前健康**的源（即未在 cooldown 中），按 `Priority` 升序排队。若无健康源，直接放出全部源（兜底，避免全冷却时彻底无响应）。
2. 启动**队首源**的 `FetchAsync`，同时启动 `hedgeInterval` 计时器。
3. **任意源 `FetchAsync` 成功** → 取消其它进行中的源 (`linkedCts.Cancel()`) → 在结果上写入 `DataSource = source.Name` → 返回 `(Result, SourceName)`。
4. **任意源 `FetchAsync` 失败** → 调用 `registry.RecordFailure(sourceName, exception)` → 不立即结束总调用，继续等其它在跑的源 / 等下一个 hedge 时刻。
5. **`hedgeInterval` 到期且仍有未启动的下一个源** → 启动下一个源、重置 hedge 计时器（即「阶梯式发出」）。
6. **所有源都失败** → 收集每个源的最末异常，抛 `HedgedExecutionException`（继承 `AggregateException`），消息中包含 `[source=tencent: HttpRequestException ...]` 列表。
7. 全程响应外层 `cancellationToken`：用户取消 → 取消全部进行中的源 → 抛 `OperationCanceledException`。
8. 单源内部重试由 `RetryPolicy` 在 `FetchAsync` 内自行完成，对 Runner 透明（即 Runner 只看到「最终成功 / 最终失败」）。

`StreamingHedgedRunner` 算法相同，但额外保证：**只让胜出源的 Stream 流出**，其余源若已开始下载则取消并 Dispose 其响应流，避免两份 Stream 同时持有句柄。

### 3.4 SourceHealthRegistry 规则

| 事件 | 行为 |
| --- | --- |
| 单次成功 | 重置该源连续失败计数 = 0；清空 cooldown |
| 单次失败 | 该源连续失败计数 +1；记录最后失败时间与异常类型 |
| 连续失败 ≥ 3 | 标记 `cooldownUntil = now + 30s`；调度阶段直接跳过该源 |
| cooldown 到期 | 自动恢复为「健康」，下一次调度参与排队（视为探测） |
| 探测期内首次成功 | 立即清空连续失败计数（恢复为常规健康） |
| 探测期内再次失败 | 立刻续 30s cooldown（不要求再连续失败 3 次） |

实现要点：

- 全部状态保存在进程内 `ConcurrentDictionary<string, SourceHealth>`。
- `SourceHealth` 字段：`int ConsecutiveFailures`、`DateTimeOffset? CooldownUntil`、`string? LastErrorType`、`DateTimeOffset? LastSuccessAt`。
- Registry 只存状态，**不**主动做后台探测（避免引入 Timer / BackgroundService 增加 SDK 复杂度），探测靠下一次真实用户调用顺带完成。

### 3.5 源链与对冲间隔配置

| 数据类型 | 源链 (P0 → P2) | Hedge 间隔 | 单源重试 |
| --- | --- | --- | --- |
| 实时行情 | 腾讯 → 新浪 → 东方财富 | 500ms | 1 次 |
| 历史 K 线 | 东方财富 → 腾讯 | 500ms | 2 次（1s / 2s） |
| 财务报表（资产负债 / 利润 / 现金流） | 东方财富 → 新浪 | 800ms | 2 次（1s / 2s） |
| 研报 / 十大股东 / 龙虎榜 / 宏观 | 东方财富（唯一） | — | 3 次（1s / 2s / 4s） |
| 公告 / PDF | 巨潮（唯一） | — | 3 次（1s / 2s / 4s）+ form-urlencoded + `Accept-Encoding: gzip, deflate` |

说明：

- 「唯一源」类目不走 Hedge，直接由 `RetryPolicy` 单源指数退避。
- 巨潮 PDF 接口对 Header 敏感（实测必须显式带 `Accept-Encoding: gzip, deflate`、`Content-Type: application/x-www-form-urlencoded`），此为硬约束。

### 3.6 HttpClient 生命周期

- `HttpDataClient` 在 SDK 内部以**单例**持有一个 `HttpClient`（避免 DNS 不刷新与 socket 耗尽两难，依赖 `SocketsHttpHandler.PooledConnectionLifetime = TimeSpan.FromMinutes(2)`）。
- 默认超时：
  - **API 请求**：`10s`
  - **PDF 流式下载**：`60s`
  - 由**请求级 `CancellationTokenSource`** 实施，不依赖 `HttpClient.Timeout`，便于 Hedge 取消。
- 默认 Header：
  - `User-Agent: Baostock.NET/1.2.0`
  - `Accept-Encoding: gzip, deflate`
- 各源在自身 `FetchAsync` 内补充：
  - 新浪：`Referer: https://finance.sina.com.cn/`
  - 东方财富：必要时补 `Referer: https://quote.eastmoney.com/`
  - 巨潮：`Content-Type: application/x-www-form-urlencoded`

## 5. 关键风险与应对

| 风险 | 触发场景 | 应对 |
| --- | --- | --- |
| `TreatWarningsAsErrors=true` 阻塞编译 | 新增源代码引入 nullable / unused / async-without-await 警告 | Sprint 1 起每个 PR 必须本地 `dotnet build -warnaserror` 通过；新增公共 API 必须带 XML 注释（`CS1591`） |
| `HedgedRunner` 取消语义错乱 | linkedCts 被 Dispose 后再 Cancel 抛 `ObjectDisposedException`；非胜出源持有的 `HttpResponseMessage` 未释放 | 单测覆盖：胜出源返回后 100ms 内验证未胜出源的 `HttpResponseMessage` 已 Dispose；用 `try/catch (ObjectDisposedException)` 吞掉 race 末端的取消异常 |
| Live 测试不稳定 | 上游接口偶发限流 / DNS 抖动 / 节假日休市 | Live 集成测试默认跑（用户决定），失败信息中区分「我方代码错」vs「上游瞬时不可用」；CI 上发红就发红，不掩盖 |
| PDF Stream 生命周期 | 用户忘记 `await using`，导致 socket 不归还 | `AnnouncementPdf` 实现 `IAsyncDisposable` 并在 README 用 `await using` 示例；`DownloadToFileAsync` 内部完成后强制 Dispose 自身 Stream |
| `companyType` 缓存陈旧 | 公司发生类型变更（极罕见，如转板） | 24h TTL；提供内部 `ClearCompanyTypeCache(code)` 用于异常排障，但不暴露公开 API |
| `CodeFormatter` 边界 | 北交所 `BJ` 代码、ST/退市股、指数代码（`SH000001`）等被混用 | 单测覆盖：`SH600519` / `SZ000001` / `BJ430047` / `SH000001`(指数) 全通过；指数与个股的区分**不在 CodeFormatter 内做**，由具体 API 自行判断（避免 Formatter 引入业务语义） |
| SemVer 语义 | 既然有 BREAKING，必须升 major？ | 项目处于 0.x → 1.x 早期阶段，且与上游 baostock 同样曾发生破坏性变更；本次按 minor 升（1.1.x → 1.2.0），但 CHANGELOG 顶部 `⚠️ BREAKING` 显式声明，README 同步迁移指引。后续 v2.0.0 视情况再正式升 major。 |
| 单例 HttpClient 长连陈旧 | DNS 切换后旧连接打不通 | `SocketsHttpHandler.PooledConnectionLifetime = 2min` |
| 巨潮 Header 缺失导致 403/415 | 漏带 `Content-Type: application/x-www-form-urlencoded` 或 `Accept-Encoding` | `CninfoAnnouncementSource.FetchAsync` 内固定写死，**不**依赖 `HttpDataClient` 的默认 Header 兜底 |
| RetryPolicy 单源 timeout 一律不重试 | v1.2.0 范围内 OperationCanceledException 一律不重试，单源 timeout 完全依赖 Hedged Runner 切备源容错；v1.2.1 计划区分 outer ct 取消 vs inner timeout，引入"timeout 视为可重试"语义。**调用方编写单源场景代码时必须自行加重试。** |

## 6. 测试策略

### 6.1 解析层（纯单测，不打外网）

- 每个 `IDataSource` 实现都必须有对应的解析层单测。
- 输入：`tests/Baostock.NET.Tests/Fixtures/{source}/{api}/sample-*.json`（或 `.html`、`.pdf` 头若干字节）。
- 这些 fixture 由 Sprint 0 的实测脚本生成、人工抽检后入库。
- 单测断言：解析出的模型字段值与 fixture 一致；`DataSource` 字段写入正确。

### 6.2 Runner / Health（fake source 单测）

- 见 S1.3。

### 6.3 集成测试（Live）

- **默认跑**（用户拍板）。
- 标记 `[Trait("Category", "Live")]`，但 CI 不排除（CI 上发红就发红，不掩盖真实回归）。
- 集成测试只覆盖「能拉到、字段非空、`DataSource` 非空」的最小契约，**不**断言具体值（避免行情波动导致测试漂移）。

### v1.2.0 已知语义边界
- HttpDataClient 的 timeout 覆盖 header + body 全过程（v1.2.0 已修）
- RetryPolicy 对 OperationCanceledException 一律不重试（含 inner timeout），见第 5 章末行
- StreamingHedgedRunner 胜出后中途断流不补救（注释 TODO 留 v1.3）
- StreamingHedgedRunner.StartNext 同步异常时 enumerator 资源轻微泄漏（v1.2.1 修）

### 6.4 用户验收

- 每个 Sprint 完成由 **User Representative Agent** 至少 **2 轮**真实交易员视角验收：
  - 第 1 轮：盘中跑（验证实时行情可用性、延迟）
  - 第 2 轮：盘后跑（验证 K 线收盘后字段稳定、财务报表 / PDF / 研报等不依赖盘口的能力）
